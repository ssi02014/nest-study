# 챕터 간 일관성 유지 규칙 (Consistency Rules)

병렬 에이전트 작업 특성상 챕터마다 메서드명·필드명·엔드포인트가 달라지는 문제가 반복 발생했다.
챕터를 작성하거나 수정할 때 이 파일을 반드시 참조한다.

---

## 1. 작성/수정 전 선행 검토 의무

챕터를 작성하거나 수정하기 전에 반드시 다음을 확인한다:

- **이전 챕터 코드 확인**: 해당 챕터가 이어받는 이전 챕터 파일을 Read로 읽고, 메서드명·DTO 필드·엔드포인트 경로를 파악한다
- **grep으로 사용처 확인**: 변경할 식별자(메서드명, 필드명 등)가 다른 챕터에서도 쓰이는지 먼저 grep한다

```bash
# 예시: 특정 메서드명이 어느 챕터에서 쓰이는지 확인
grep -rn "메서드명" docs/
```

---

## 2. 수정 후 교차 검증 체크리스트

챕터 수정 완료 후 반드시 아래 항목을 확인한다:

- [ ] 변경된 메서드명·필드명을 `grep -rn "변경전이름" docs/`로 검색 → 남은 불일치 없는지 확인
- [ ] 수정 챕터의 **다음 챕터(N+1)** 가 수정 내용을 올바르게 이어받는지 확인
- [ ] DTO 필드 추가/제거 시 해당 DTO를 사용하는 모든 챕터(curl 예시 포함) 확인
- [ ] HTTP 메서드·상태 코드 일관성 확인
- [ ] 타입 불일치 비교(`number vs string`) 확인

---

## 3. 병렬 에이전트 작업 후 검증 단계 의무화

복수의 에이전트로 챕터를 병렬 작성/수정한 경우, **완료 후 반드시 교차 검증 에이전트를 별도로 실행**한다. 교차 검증 없이 작업 완료로 보고하지 않는다.

교차 검증 에이전트 지시 예시:

```
docs/ 전체에서 아래 항목을 검증해주세요:
1. Service 메서드명이 챕터 전체에서 일관되게 사용되는지
2. DTO 필드가 챕터 흐름에 맞게 추가/제거되었는지
3. HTTP 상태 코드(DELETE 204 등)가 일관적인지
4. 챕터 N → N+1 코드 연속성 (메서드명·DTO 필드·경로 일치 여부)
```

---

## 4. 규칙 파일 갱신 원칙

새로운 일관성 이슈가 발견되었을 때, 이 파일에 아직 명시되어 있지 않다면 **반드시 이 파일을 먼저 업데이트한 후 작업을 계속한다.**

- 수정 작업 중 새로운 패턴의 실수를 발견한 경우 → `§5 자주 발생한 실수 유형`에 추가
- 기존 체크리스트로 잡아내지 못한 종류의 불일치가 생긴 경우 → `§2 수정 후 교차 검증 체크리스트`에 항목 추가
- 에이전트 작업 방식 자체에서 구조적 문제가 발견된 경우 → `§3 병렬 에이전트 작업 후 검증` 교차 검증 지시 예시에 추가

> 이 규칙 파일이 실제 발생한 문제를 반영하지 못하면 같은 실수가 반복된다.

---

## 5. 자주 발생한 실수 유형 (히스토리)

반복된 실수를 기록한다. 같은 실수를 두 번 하지 않기 위해 참조한다.

| 실수 유형 | 구체적 사례 | 올바른 처리 |
|-----------|-------------|-------------|
| 메서드명 불일치 | 같은 메서드를 챕터마다 다른 이름으로 호출 | 작성 전 이전 챕터를 Read로 확인 |
| 타입 불일치 비교 | `c.postId === postId` (number vs string) | 타입 변환 후 비교 |
| 상태 코드 오류 | E2E 테스트에서 DELETE `.expect(200)` | 컨트롤러의 `@HttpCode` 확인 후 맞춤 |
| 엔티티 필드 오참조 | Entity에 없는 필드명 사용 | 작성 전 Entity 정의 확인 |
| `import type` 남용 | DI 주입 클래스에 `import type` 사용 | 주입받는 클래스는 반드시 일반 `import` |
| 병렬 작업 후 미검증 | 에이전트 작성 후 교차 검증 없이 완료 처리 | 완료 전 검증 에이전트 실행 |
